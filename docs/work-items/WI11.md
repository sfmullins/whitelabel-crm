# WI11 — Extension Platform

WI11 provides an upgrade-safe declarative extension layer on the merged WI10 API, event and security foundation.

## Delivery status

Implementation is complete on `WI11-Extension-Platform` and presented for review in PR #14. WI12 must not begin until WI11 is reviewed and merged.

## Delivered scope

- strict versioned JSON manifest and package schemas;
- application-version compatibility checks;
- explicit per-install capability approval;
- deterministic canonical SHA-256 manifest checksums;
- optional detached Ed25519 signature verification;
- bounded extension, release, contribution, binding, migration, asset, install-attempt and data-action registries;
- verified pre-migration SQLite backups for schema-affecting install, upgrade and purge operations;
- transactional installation and upgrade, preserving the previous active release after failure;
- non-destructive enable and disable controls;
- separate retired-contribution state so package upgrades do not revive removed resources;
- namespaced custom fields and custom entities;
- safe custom-field/entity upgrade rules prohibiting identity or type mutation;
- system-managed bridging of pre-WI11 custom fields and objects;
- protection against direct deletion of extension-owned definitions through legacy administration APIs;
- runtime registries for forms, views, navigation, themes, reports, workflow templates, event subscriptions, localisation, custom metadata and static assets;
- dynamic extension navigation and theme selection in the React shell;
- generic declarative extension workspaces for forms, views, reports, workflows and assets;
- extension reports executed through the existing `ReportingRepository` rather than a parallel reporting engine;
- explicit workflow-template instantiation into disabled WI7 workflow definitions for human review;
- exact-match asset catalogue validation, strict base64 decoding, size and SHA-256 verification, safe local storage and verification again when served;
- active-content restrictions for packaged SVG and structural validation for packaged JSON;
- metadata/history export and extension-owned data export as distinct operations;
- disabled-only, exact-confirmation data purge with a recovery backup;
- exact-confirmation release recovery through the existing backup manager;
- explicit `extensions.read` and `extensions.manage` administration permissions while ordinary CRM users may read active runtime metadata;
- immutable WI10 platform events for install, upgrade, enable and disable operations;
- audit coverage for installation, lifecycle, export, purge, workflow instantiation and recovery requests;
- administration UI for validation, capability approval, installation, upgrades, enable/disable, exports, purge and recovery;
- permanent `wi11:smoke` and backend lifecycle/runtime tests;
- named CI stages and failure-only build/test diagnostics.

## Security constraints

Extensions cannot contain or execute JavaScript, SQL, shell commands or renderer bundles. They have no direct database or credential access. All schema changes are limited to supported declarative custom-field and custom-entity operations.

Package installation does not activate workflow automation. A workflow template must be explicitly instantiated, is created disabled and remains governed by the existing allow-listed workflow engine and policy controls. Disabling an extension also disables workflows instantiated from that extension.

Static assets are bounded to 5 MB each, require declared media types, size and SHA-256 digest, and must exactly match the supplied package files. Paths cannot traverse package directories. SVG files containing scripts, event handlers, JavaScript URLs, foreign objects or external HTTP references are rejected.

Data purge is separate from extension disable and from export. It requires:

1. a non-system extension;
2. disabled extension state;
3. the exact phrase `PURGE <package-key>`;
4. a successful backup before deletion begins.

Release recovery restores a complete SQLite backup and therefore requires the exact phrase `RESTORE <package-key> <version>`. The desktop application should be restarted after recovery.

## Package outline

```json
{
  "manifest": {
    "formatVersion": 1,
    "packageKey": "vendor.case-management",
    "name": "Case Management",
    "version": "1.0.0",
    "application": { "minVersion": "1.0.0" },
    "capabilities": ["custom_fields", "navigation", "static_assets"],
    "contributions": {
      "customFields": [],
      "customEntities": [],
      "forms": [],
      "views": [],
      "navigation": [],
      "themes": [],
      "reports": [],
      "workflowTemplates": [],
      "eventSubscriptions": [],
      "localisations": [],
      "assets": [
        {
          "key": "help",
          "path": "help/index.json",
          "mediaType": "application/json",
          "sha256": "<64-character digest>",
          "sizeBytes": 123
        }
      ]
    }
  },
  "files": [
    {
      "path": "help/index.json",
      "contentBase64": "<base64 payload>"
    }
  ]
}
```

An optional `signature` object may contain an Ed25519 public key and detached signature over the canonical manifest JSON.

## Deliberate exclusions

- executable or in-process plugins;
- arbitrary SQL, JavaScript, shell or renderer bundles;
- direct extension database credentials;
- automatic workflow activation;
- destructive schema migrations or automatic definition deletion;
- remote marketplace installation;
- arbitrary report queries or formulas;
- arbitrary route components supplied by packages;
- automatic data deletion during disable, upgrade or uninstall.
