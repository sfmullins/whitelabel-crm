# WI12 — Instance Onboarding, Provisioning and Deployment Profiles

## Objective

WI12 inserts a mandatory business-instance configuration and publication layer before final packaging. A business configures, validates, previews and publishes its CRM once; employee clients then bind to that approved instance rather than creating unrelated databases on every machine.

WI12 is not the final packaging work item. Final installers, release provenance, SBOM publication and cross-platform release certification move to WI13.

## Deployment topology

### Managed business instance

The recommended employee topology:

- one authoritative backend and SQLite database;
- employees connect through a branded desktop client or supported browser;
- the managed Electron client verifies a signed deployment profile;
- the client does not start an embedded business database;
- backups, migrations and recovery are administered centrally.

### Standalone instance

- one embedded backend and local SQLite database;
- deliberately isolated from other installations;
- not represented as a shared multi-user deployment;
- retains the existing local-first operating model.

A configured production database must never be cloned onto employee machines as a substitute for a managed deployment. That would create divergent authoritative copies, conflicting audit histories and unsafe credential distribution.

## Delivered architecture

### Canonical configuration

`crm_instances` and `instance_configuration_revisions` provide a versioned configuration registry. Revisions use explicit `draft`, `published`, `superseded` and `rolled_back` lifecycle states. Only drafts are editable.

Existing `settings` values are preserved as the first published revision during migration. Publication mirrors compatible identity, branding, locale and financial settings back into the existing settings boundary so current application surfaces remain canonical.

### Readiness

The readiness engine returns stable machine-readable checks with:

- category;
- required or recommended severity;
- passed, warning, failed or not-applicable status;
- explanation;
- remediation;
- section link;
- evidence.

Required failures block publication. Recommended warnings remain visible without being misrepresented as hard failures.

### Publication

Publication:

1. validates the complete configuration contract;
2. records readiness evidence;
3. creates a verified pre-publication backup;
4. freezes the draft revision;
5. creates a deterministic deployment profile;
6. calculates its SHA-256 checksum;
7. signs it with an instance Ed25519 key held in the credential vault;
8. atomically activates the new revision;
9. supersedes the prior publication;
10. opens the next editable draft;
11. records immutable audit and platform events.

A failed publication leaves the previous revision active.

### Deployment profile

The signed profile contains safe deployment data only:

- immutable instance identity;
- configuration revision;
- managed or standalone topology;
- approved managed origin;
- business display identity;
- bounded branding tokens and embedded assets;
- locale and business terminology;
- enabled capability identifiers;
- minimum client version;
- publication timestamp.

It excludes passwords, sessions, API tokens, backup passwords, OAuth secrets, storage credentials, private signing keys and the live CRM database.

### Employee enrolment

Employee activation uses short-lived one-time enrolment tokens:

- cryptographically random;
- hash-only at rest;
- instance- and user-bound;
- device-limited;
- revocable;
- audited;
- exchanged for a user-scoped session and device registration.

The raw token is returned once and is recursively redacted from request/response audit payloads.

### Managed Electron client

A managed client:

- loads the packaged bootstrap profile;
- verifies checksum and Ed25519 signature;
- enforces the minimum client version;
- requires HTTPS outside explicit development mode;
- binds to one instance ID, signing key and instance URL;
- rejects profile tampering, URL replacement and downgrade;
- refreshes only from the bound instance;
- does not start the embedded backend or local authoritative database;
- disables local backup/restore selection because recovery is central.

## User experience

The former three-step settings form is replaced by a persistent onboarding workspace containing:

- answer-first readiness dashboard;
- managed versus standalone topology selection;
- business identity;
- branding studio and live employee preview;
- locale and regional settings;
- business terminology;
- organisation structure and CRM operating model;
- communications and financial defaults;
- security and recovery controls;
- employee rollout and one-time enrolment;
- review, approval, publication and profile download.

Draft changes autosave and do not affect employees before publication.

## Security boundaries

New permissions:

- `onboarding.read`;
- `onboarding.manage`;
- `deployment.publish`;
- `devices.manage`.

Only two endpoints are deliberately pre-authentication:

- signed public-profile discovery;
- one-time enrolment redemption.

All other onboarding operations require an authenticated identity and explicit permission. Public endpoints remain subject to origin policy, rate limiting, schema validation and audit redaction.

## Permanent gates

```bash
npm run onboarding:verify
npm run managed-client:smoke
npm run deployment:verify
npm run wi12:smoke
```

`npm run ci:verify` includes WI12 after every WI4–WI11 regression gate.

## Deferred to WI13

- final Windows installer publication;
- final Linux installer publication;
- public container release;
- release signing services;
- published SBOMs and provenance;
- stable release promotion;
- installed-artifact cross-platform certification.

WI13 consumes the WI12 deployment profile. It must not bypass the WI12 publication gate.
