# 5. Canonical conventions

Use these conventions unless the existing repository has a stronger compatible convention.

## IDs

Use UUID strings generated using the repository’s current canonical mechanism:

```ts
randomUUID()
```

IDs are:

- generated by the backend;
- immutable;
- never accepted in create request bodies;
- never accepted in update request bodies.

Route parameters representing IDs must be validated as UUIDs before repository access.

## Timestamps

Use the current timestamp storage convention:

```ts
new Date().toISOString()
```

The database stores timestamps as ISO-8601 UTC text.

The backend owns:

- `createdAt`;
- `updatedAt`;
- `archivedAt`.

Clients must not be able to submit or overwrite these fields.

## Date-only values

Engagement `startDate` and `endDate` are calendar dates, stored as:

```text
YYYY-MM-DD
```

Validate both the format and actual calendar validity. Do not accept impossible dates such as `2026-02-31`.

Because valid ISO calendar dates are lexicographically sortable, date ordering may be compared after strict validation.

Do not convert date-only fields through local timezone-dependent JavaScript date formatting.

## Optional database values

Use SQL `NULL` for absent optional values.

For API contracts:

- omitted update field = retain the current value;
- explicit `null` = clear an optional field;
- whitespace-only optional strings should normalize to `null`;
- required strings must reject `null`, empty strings and whitespace-only strings.

Do not use an update schema produced by blindly calling `.partial()` on a full response/entity schema, because that would permit immutable fields.

## Boolean values

Use the established Drizzle SQLite boolean representation:

```ts
integer('is_primary', { mode: 'boolean' })
```

## Hard deletion

Do not expose hard-delete endpoints for these entities.

Foreign keys must not cascade-delete organisations, contacts or engagements.

Use `RESTRICT` or the equivalent established SQLite/Drizzle behaviour for business-record relationships.

# 6. Domain invariants

The following rules are mandatory and must be enforced in application code and tests. Add database constraints where practical and compatible.

## Organisation invariants

- `name` is required after trimming.
- Archived organisations are excluded from normal lists and normal lookup operations.
- New contacts and engagements cannot be added to an archived organisation.
- Archived organisations cannot be edited through the ordinary update operation.
- Archiving an organisation does not delete or automatically archive its contacts or engagements.
- Repeated archive requests are idempotent.
- Organisation archive must not alter legacy customer records.

## Contact invariants

- Every contact belongs to exactly one organisation.
- `organisationId` is immutable after creation.
- At least one of `firstName`, `lastName` or `email` must remain populated after trimming.
- Email is optional but valid when supplied.
- Email is not globally unique.
- A contact cannot be primary if it is inactive or archived.
- At most one contact per organisation may simultaneously satisfy:

```text
isPrimary = true
status = active
archivedAt IS NULL
```

- Setting one eligible contact as primary must clear `isPrimary` from every other non-archived contact in that organisation in the same transaction.
- Setting `isPrimary` to `false` affects only that contact.
- Changing a primary contact to `inactive` must also set its `isPrimary` value to `false` atomically.
- Archiving a primary contact must also set its `isPrimary` value to `false` atomically.
- Archiving a contact must not delete the contact or mutate historical engagement records.
- An ordinary update cannot move a contact between organisations.
- Archived contacts cannot be edited by the ordinary update operation.

## Engagement invariants

- Every engagement belongs to exactly one organisation.
- `organisationId` is immutable after creation.
- `primaryContactId` is optional.
- When assigned, the primary contact must:
  - exist;
  - belong to the same organisation;
  - have `status = active`;
  - have `archivedAt = null`.
- An engagement can exist without a primary contact.
- `name` is required after trimming.
- `endDate`, when supplied, cannot precede `startDate`.
- Changing `startDate` or `endDate` must validate the resulting complete date pair, not merely the incoming patch fields.
- Archived engagements cannot be edited through the ordinary update operation.
- Archiving an engagement does not delete it.
- Archiving a contact later does not erase that contact ID from historical engagement records. Eligibility is checked when assigning or changing the primary contact.
- An ordinary update cannot move an engagement between organisations.

## Suggested validation helpers

The following illustrates the intended behaviour. Adapt naming and placement to the repository; do not copy it blindly if a shared helper pattern already exists.

```ts
import { z } from 'zod';

const requiredTrimmedString = (label: string) =>
  z.string()
    .trim()
    .min(1, `${label} is required`);

const nullableTrimmedString = z.preprocess(
  (value) => {
    if (value === null || value === undefined) return value;
    if (typeof value !== 'string') return value;

    const trimmed = value.trim();
    return trimmed === '' ? null : trimmed;
  },
  z.string().nullable().optional(),
);

const isoDateSchema = z.string().superRefine((value, ctx) => {
  if (!/^\d{4}-\d{2}-\d{2}$/.test(value)) {
    ctx.addIssue({
      code: z.ZodIssueCode.custom,
      message: 'Date must use YYYY-MM-DD',
    });
    return;
  }

  const [year, month, day] = value.split('-').map(Number);
  const date = new Date(Date.UTC(year, month - 1, day));

  if (
    date.getUTCFullYear() !== year ||
    date.getUTCMonth() !== month - 1 ||
    date.getUTCDate() !== day
  ) {
    ctx.addIssue({
      code: z.ZodIssueCode.custom,
      message: 'Date is not a valid calendar date',
    });
  }
});
```

export const OrganisationCreateSchema = z.object({

## Contact identifying-field rule

Implement the rule on both create and update results.

For create:

```ts
const ContactFieldsSchema = z.object({
  organisationId: z.string().uuid(),
  firstName: nullableTrimmedString,
  lastName: nullableTrimmedString,
  jobTitle: nullableTrimmedString,
  email: /* nullable, trimmed, lowercased and validated */,
  phone: nullableTrimmedString,
  isPrimary: z.boolean().default(false),
  status: ContactStatusSchema.default('active'),
});

export const ContactCreateSchema = ContactFieldsSchema
  .strict()
  .superRefine((value, ctx) => {
    if (!value.firstName && !value.lastName && !value.email) {
      ctx.addIssue({
        code: z.ZodIssueCode.custom,
        path: ['firstName'],
        message: 'At least one of firstName, lastName or email is required',
      });
    }

    if (value.isPrimary && value.status !== 'active') {
      ctx.addIssue({
        code: z.ZodIssueCode.custom,
        path: ['isPrimary'],
        message: 'An inactive contact cannot be primary',
      });
    }
  });
```

For update, schema validation alone cannot know whether the resulting stored record still has an identifying field. The service must:

1. load the current contact;
2. apply the validated patch;
3. validate the resulting complete contact;
4. reject a patch that would clear all identifying fields.

# 18. Primary-contact transaction

The primary-contact invariant must be implemented as one database transaction.

The transaction must cover both:

1. clearing other eligible primary contacts;
2. writing the candidate contact’s new state.

The implementation should conceptually resemble:

```ts
database.transaction((tx) => {
  tx.update(contacts)
    .set({
      isPrimary: false,
      updatedAt: now,
    })
    .where(
      and(
        eq(contacts.organisationId, organisationId),
        ne(contacts.id, contactId),
        eq(contacts.isPrimary, true),
        isNull(contacts.archivedAt),
      ),
    )
    .run();

  tx.update(contacts)
    .set({
      ...patch,
      isPrimary: true,
      updatedAt: now,
    })
    .where(
      and(
        eq(contacts.id, contactId),
        eq(contacts.organisationId, organisationId),
        eq(contacts.status, 'active'),
        isNull(contacts.archivedAt),
      ),
    )
    .run();
});
```

Adapt this to the installed Drizzle/better-sqlite3 API.

Important requirements:

- do not clear the old primary before validating the candidate;
- validate candidate existence, organisation, status and archive state first;
- execute the clear and set writes inside one transaction;
- ensure an exception rolls back both writes;
- ensure the candidate update affected exactly one row;
- protect the invariant with the partial unique index where supported;
- translate a unique-constraint race into a safe conflict response rather than exposing SQLite text.

Creating a new contact with `isPrimary: true` must use the same transaction semantics.

Changing a contact to inactive or archiving it must clear its own `isPrimary` flag in the same transaction.

Do not use a read-then-write sequence without a transaction.

## Assigning an engagement primary contact

Centralize this validation in one private or reusable service method, conceptually:

```ts
private async validatePrimaryContact(
  organisationId: string,
  contactId: string | null | undefined,
): Promise<void> {
  if (contactId === null || contactId === undefined) return;

  const contact = await this.contacts.getById(contactId, {
    includeArchived: true,
  });

  if (!contact) {
    throw new NotFoundError('Primary contact not found');
  }

  if (contact.organisationId !== organisationId) {
    throw new ConflictError(
      'Primary contact must belong to the engagement organisation',
    );
  }

  if (contact.archivedAt || contact.status !== 'active') {
    throw new ConflictError(
      'Primary contact must be active and non-archived',
    );
  }
}
```

# 20. Domain errors and safe HTTP mapping

Do not expose raw SQLite, Drizzle or stack-trace details.

Add a minimal typed operational-error mechanism. For example:

```ts
export class AppError extends Error {
  constructor(
    public readonly statusCode: number,
    message: string,
    public readonly code: string,
    public readonly details?: unknown,
  ) {
    super(message);
  }
}

export class NotFoundError extends AppError {
  constructor(message: string) {
    super(404, message, 'NOT_FOUND');
  }
}

export class ValidationError extends AppError {
  constructor(message: string, details?: unknown) {
    super(400, message, 'VALIDATION_ERROR', details);
  }
}

export class ConflictError extends AppError {
  constructor(message: string) {
    super(409, message, 'CONFLICT');
  }
}
```

Adapt this to any existing error utility found during inspection.

The global error middleware should return operational messages only for known application errors.

Unknown failures must return a generic response:

```ts
if (err instanceof AppError) {
  return res.status(err.statusCode).json({
    error: err.code,
    message: err.message,
    ...(err.details === undefined ? {} : { details: err.details }),
  });
}

console.error('Unhandled Server Error:', err);

return res.status(500).json({
  error: 'INTERNAL_SERVER_ERROR',
  message: 'An unexpected error occurred',
});
```

Do not return:

- SQL statements;
- constraint names as the only explanation;
- stack traces;
- arbitrary `err.message` for unknown errors;
- database file paths.

Keep this hardening small and compatible with existing routes. Do not redesign every existing error response unless needed to prevent internal-error leakage.
