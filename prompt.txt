# 4. Critical defect: broken primary-contact transaction

The current implementation in:

```text
backend/src/infrastructure/database/repositories/ContactRepository.ts
```

runs a synchronous `better-sqlite3` transaction callback but calls the asynchronous repository method `this.update()` inside it.

The current pattern is substantively similar to:

```ts
sqlite.transaction(() => {
  sqlite.prepare('update contacts ...').run(...);

  this.update(id, {
    ...patch,
    isPrimary: true,
  });
})();
```

This is unsafe.

Because `this.update()` is an `async` function, failures become rejected promises rather than synchronous exceptions visible to the transaction callback.

The transaction may therefore:

1. clear the existing primary contact;
2. fail to set the candidate contact as primary;
3. commit the first update;
4. leave the organisation without its previous primary.

This violates the required atomicity and rollback guarantee.

## Required correction

Both writes must execute synchronously inside the same database transaction callback.

Do not call an `async` repository method from inside a synchronous `better-sqlite3` transaction.

Use one of these valid approaches:

### Preferred approach

Create private synchronous repository helpers that accept the active transaction connection or use the same raw SQLite connection synchronously.

For example:

```ts
private updateContactSync(
  connection: SqliteConnection,
  id: string,
  patch: ContactUpdate,
  now: string,
): ContactUpdateResult {
  // Build and execute the update synchronously.
}
```

Then:

```ts
const transaction = sqlite.transaction(() => {
  const candidate = sqlite.prepare(`
    SELECT *
    FROM contacts
    WHERE id = ?
      AND archived_at IS NULL
  `).get(id) as ContactRow | undefined;

  if (!candidate) {
    throw new ContactPrimaryEligibilityError('Contact not found');
  }

  const resultingStatus = patch.status ?? candidate.status;

  if (resultingStatus !== 'active') {
    throw new ContactPrimaryEligibilityError(
      'An inactive contact cannot be primary',
    );
  }

  const now = new Date().toISOString();

  sqlite.prepare(`
    UPDATE contacts
    SET is_primary = 0,
        updated_at = @now
    WHERE organisation_id = @organisationId
      AND id <> @id
      AND archived_at IS NULL
      AND is_primary = 1
  `).run({
    organisationId: candidate.organisation_id,
    id,
    now,
  });

  const result = sqlite.prepare(`
    UPDATE contacts
    SET
      status = @status,
      is_primary = 1,
      updated_at = @now
    WHERE id = @id
      AND archived_at IS NULL
  `).run({
    id,
    status: resultingStatus,
    now,
  });

  if (result.changes !== 1) {
    throw new ContactPrimaryEligibilityError(
      'Primary contact could not be updated',
    );
  }
});

transaction();
```

The actual implementation must support every supplied mutable field, not just status and `isPrimary`.

### Drizzle transaction alternative

A Drizzle transaction is acceptable if the installed SQLite driver executes it synchronously and affected-row counts can be verified reliably.

Do not use an asynchronous callback with `better-sqlite3` unless the installed Drizzle adapter explicitly supports and safely waits for it.

## Required guarantees

The corrected implementation must prove that:

- candidate validation happens before clearing the existing primary;
- the old primary is not cleared if the candidate is invalid;
- clearing the old primary and setting the new primary are one transaction;
- any exception rolls back both writes;
- the candidate update affects exactly one row;
- a partial unique-index conflict is translated into a controlled domain conflict;
- no raw SQLite message reaches the client;
- creating a new primary contact follows the same atomic principle;
- making an existing inactive contact active and primary in one update works correctly;
- making an active primary contact inactive clears its primary status atomically;
- archiving a primary contact clears its primary status atomically.

# 5. Correct primary-contact state-transition semantics

The current service mutates this contradictory patch:

```json
{
  "status": "inactive",
  "isPrimary": true
}
```

into:

```json
{
  "status": "inactive",
  "isPrimary": false
}
```

That is incorrect.

The API must not silently reinterpret contradictory client intent.

## Required behaviour

### Reject this request

```json
{
  "status": "inactive",
  "isPrimary": true
}
```

Return a controlled `409 Conflict` or `400 Validation Error`, consistently with the domain-error policy.

Preferred response:

```json
{
  "error": "CONFLICT",
  "message": "An inactive contact cannot be primary"
}
```

### Accept this request

For an existing inactive contact:

```json
{
  "status": "active",
  "isPrimary": true
}
```

This must atomically:

1. change the contact to active;
2. clear any other primary contact in the organisation;
3. set the candidate as primary.

Do not validate eligibility solely against the contact’s old stored status.

Validation must use the resulting merged state:

```ts
const merged = {
  ...current,
  ...patch,
};

if (merged.isPrimary && merged.status !== 'active') {
  throw new ConflictError('An inactive contact cannot be primary');
}
```

The repository transaction must also apply the resulting state atomically.

### Inactivation rule

For:

```json
{
  "status": "inactive"
}
```

the final state must be:

```text
status = inactive
isPrimary = false
```

This update must be atomic.

### Explicit un-primary rule

For:

```json
{
  "isPrimary": false
}
```

only that contact should lose primary status.

Do not automatically promote another contact.

# 6. Eliminate raw generic primary-contact errors

The repository currently throws:

```ts
throw new Error('CONTACT_PRIMARY_NOT_ELIGIBLE');
```

This becomes an unknown server failure and therefore an HTTP `500`.

Do not throw generic errors for expected business conditions.

Use typed operational errors at the service boundary.

A repository may return a typed result or throw a narrowly scoped internal repository error, but the application service must translate it to:

- `NotFoundError`;
- `ConflictError`;
- `ValidationError`.

For example:

```ts
export class PrimaryContactConflictError extends Error {
  constructor(
    public readonly reason:
      | 'not_found'
      | 'archived'
      | 'inactive'
      | 'unique_conflict',
  ) {
    super(reason);
  }
}
```

Then translate:

```ts
try {
  return await this.contacts.updatePrimary(id, patch);
} catch (error) {
  if (error instanceof PrimaryContactConflictError) {
    switch (error.reason) {
      case 'not_found':
        throw new NotFoundError('Contact not found');

      case 'archived':
        throw new ConflictError('Archived contacts cannot be edited');

      case 'inactive':
        throw new ConflictError('An inactive contact cannot be primary');

      case 'unique_conflict':
        throw new ConflictError(
          'The organisation already has another primary contact',
        );
    }
  }

  throw error;
}
```

The exact design may differ, but expected domain conflicts must not become generic `500` responses.
