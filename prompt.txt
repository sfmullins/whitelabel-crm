# 1. Mission

Implement a production-safe, normalised activity and notes system for the CRM.

The current frontend appends timestamped note blocks into the legacy `customers.notes` text column and reconstructs a timeline by parsing that combined string. That is not an acceptable long-term data model.

WI3 must:

1. add a first-class `activities` domain model;
2. represent each note, call, email, meeting or other interaction as an independent record;
3. link activities to an organisation and optionally to a contact and/or engagement;
4. retain author attribution, event time, optional follow-up date and archive state;
5. migrate existing legacy customers into explicit organisation/contact mappings without deleting or rewriting the legacy customer rows;
6. import existing `customers.notes` content into distinct activity rows without losing text;
7. stop the frontend from parsing or appending combined note strings;
8. preserve all existing customers, bookings, invoices, invoice items and payments unchanged;
9. provide shared contracts, validation, persistence, services, REST endpoints, migration/backfill tests and focused frontend integration;
10. leave later work items with a stable foundation for tasks, reminders, documents, richer organisation workspaces and financial migration.

This is an implementation task, not an architecture-only proposal. Inspect the repository, implement the work, run the required validation, push the branch and open the pull request.

# 6. Canonical activity model

Implement one canonical `activities` table. A note is an activity whose `type` is `note`; do not create a second combined-note text store.

Use this logical contract:

| Field | Database | Required | Rules |
|---|---|---:|---|
| `id` | text primary key | yes | Backend-generated UUID |
| `organisationId` | text FK | yes | References `organisations.id`, immutable after create |
| `contactId` | text FK nullable | no | References `contacts.id`; when supplied it must belong to the same organisation |
| `engagementId` | text FK nullable | no | References `engagements.id`; when supplied it must belong to the same organisation |
| `type` | text | yes | Controlled value |
| `body` | text | yes | Trimmed and non-empty |
| `author` | text | yes | Trimmed human-readable attribution label |
| `occurredAt` | text | yes | ISO-8601 UTC timestamp for when the interaction occurred |
| `followUpDate` | text nullable | no | Valid calendar date in `YYYY-MM-DD` format |
| `source` | text | yes | Backend-owned controlled value: `user`, `legacy_import` or `system` |
| `sourceReference` | text nullable | no | Backend-owned idempotency/traceability key for imported/system records |
| `createdAt` | text | yes | Backend ISO timestamp |
| `updatedAt` | text | yes | Backend ISO timestamp |
| `archivedAt` | text nullable | no | Soft-archive timestamp |

## Controlled activity types

Define the values once in the shared package and derive TypeScript types from the Zod enum:

```ts

  'note',
  'call',
  'email',
  'meeting',
  'message',
  'other',
]);
```

Do not duplicate string unions in routes, services or repositories.

## Activity source values

```ts

  'user',
  'legacy_import',
  'system',
]);
```

`source` and `sourceReference` are backend-owned. They must not be accepted from ordinary create or update request bodies.

## Author handling before authentication exists

The application does not yet have user identities.

For WI3:

- `author` remains a required, stored response field;
- ordinary create requests may supply a trimmed author label;
- when omitted, the application service must use the explicit fallback `Local user`;
- imported records must use `Legacy import`;
- system-generated records, if any are required internally, must use `System`;
- do not add authentication or a users table in this work item.

This is an interim attribution label, not an authenticated identity claim. Document that limitation.

# 9. Deterministic customer mapping rules

Implement an idempotent mapping operation that can be called by the migration backfill and by customer compatibility activity routes.

For each legacy customer without a mapping:

## 9.1 Customer with a non-empty company

1. Trim the company value.
2. Apply Unicode `NFKC` normalisation.
3. Collapse internal whitespace runs to one space.
4. Create a deterministic comparison key by lowercasing the normalised value.
5. Prefix the key so namespaces cannot collide:

```text
company:<normalised-lowercase-company>
```

6. Reuse only an organisation already referenced by `legacy_organisation_mappings` for that exact key.
7. If no mapping exists, create a new organisation with:
   - `name` equal to the first original trimmed display value encountered for that key;
   - `status = active_client`;
   - `source = legacy_customer_import`;
   - no inferred legal name, website, industry, employee band, revenue band or country.
8. Do not fuzzy-match or automatically merge with manually created organisations.

## 9.2 Customer without a company

Create one dedicated imported organisation using:

```text
sourceKey = individual_customer:<customerId>
```

Use the first available non-empty label in this order:

1. trimmed full name;
2. email;
3. `Legacy customer <customerId>`.

Set:

```text
status = active_client
source = legacy_customer_import
```

Do not group unrelated individual customers into one generic organisation.

## 9.3 Imported contact

Create one contact for the mapped customer:

- copy `firstName`;
- copy `lastName`;
- copy and normalise `email` using the existing contact conventions;
- use `mobile` when non-empty, otherwise `phone`, for the single contact `phone` field;
- set `status = active`;
- set `jobTitle = null`;
- do not invent missing values;
- do not remove address, tags, alternate phone or any other values from the source customer row.

Primary-contact handling must be deterministic:

- when an imported organisation has exactly one mapped customer, its imported contact may be primary;
- when an imported company organisation has multiple mapped customers, do not arbitrarily choose one as primary;
- preserve the one-active-primary invariant established in WI2;
- never clear or replace a manually selected primary contact on an existing organisation.

Because imported organisations are not fuzzy-merged into existing manual organisations, this rule should not mutate manual CRM records.

## 9.4 Idempotency

Re-running the mapping operation must:

- return the existing mapping;
- not create duplicate organisations;
- not create duplicate contacts;
- not change legacy customer data;
- not change an existing mapping’s organisation/contact IDs;
- tolerate a partially completed previous run by either completing safely or failing with a precise integrity error.

Use transactions for multi-row mapping creation.

# 10. Legacy note import

Implement a dedicated, tested backfill that runs after the structural Drizzle migrations have completed.

The backfill must scan legacy customers, ensure each relevant customer has a CRM mapping, and import non-empty `customers.notes` content into `activities`.

## 10.1 Source formats

The current UI has produced blocks shaped like:

```text
[Note logged on <timestamp>]:
<body>
```

A customer notes field may contain:

- one well-formed block;
- several well-formed blocks separated by blank lines;
- free text before the first block;
- only unstructured free text;
- malformed markers;
- localised date strings that JavaScript can parse;
- localised date strings that JavaScript cannot parse;
- leading/trailing whitespace;
- mixed line endings.

The importer must not assume every value is perfectly formed.

## 10.2 Parsing requirements

For each customer with non-empty notes:

1. normalise line endings for parsing only;
2. identify every well-formed legacy marker and its following body;
3. import each recognised block as a separate `note` activity;
4. import any non-empty unmatched text as one or more fallback note activities so that no text is discarded;
5. retain body text faithfully apart from trimming accidental outer whitespace;
6. never silently drop malformed marker text;
7. never rewrite or clear `customers.notes`.

For a recognised timestamp:

- attempt to parse it;
- when valid, store `occurredAt` as `new Date(value).toISOString()`;
- when invalid, use `customer.updatedAt`, otherwise `customer.createdAt`;
- preserve the unparseable timestamp text in the imported body or other explicit human-readable form so that information is not lost.

For unstructured fallback text, use:

1. `customer.updatedAt` when valid;
2. otherwise `customer.createdAt` when valid;
3. otherwise the import timestamp.

All imported activities must have:

```text
type = note
author = Legacy import
source = legacy_import
organisationId = mapped organisation
contactId = mapped contact
engagementId = null
followUpDate = null
```

## 10.3 Idempotency key

Generate a deterministic `sourceReference` for each imported segment.

Use a stable scheme based on:

- legacy customer ID;
- segment ordinal;
- a SHA-256 digest of the exact imported segment before outer trimming.

For example:

```text
legacy-customer-note:<customerId>:<ordinal>:<sha256>
```

The database unique index must prevent duplicate insertion.

Re-running the backfill against unchanged source data must insert zero duplicate activities.

Do not use random UUIDs as the only import deduplication mechanism.

## 10.4 Transaction and failure semantics

Run mapping and note import in bounded transactions.

A failure must:

- roll back the affected transaction;
- leave legacy rows untouched;
- expose enough context in logs/tests to identify the customer and operation;
- never leave a mapping row pointing at missing CRM entities.

Do not swallow migration errors and continue with a falsely successful startup.

# 13. Activity invariants

Enforce these in the application service and tests.

## Organisation

- Every activity belongs to exactly one organisation.
- The organisation must exist when an activity is created.
- New activities cannot be added to an archived organisation.
- `organisationId` cannot be changed by ordinary update.
- Archiving an organisation later does not delete its historical activities.

## Contact

- `contactId` is optional.
- When supplied, the contact must exist.
- It must belong to the activity organisation.
- It must be non-archived at assignment time.
- An inactive but non-archived contact may retain or receive historical activity because status does not erase history.
- Archiving a contact later does not clear historical activity references.

## Engagement

- `engagementId` is optional.
- When supplied, the engagement must exist.
- It must belong to the activity organisation.
- It must be non-archived at assignment time.
- Proposed, active, paused, completed and cancelled engagements may retain activity history.
- Archiving an engagement later does not clear historical activity references.

## Contact and engagement together

- They must both belong to the same activity organisation.
- The contact does not need to be the engagement primary contact.

## Body and time

- Body is always trimmed and non-empty.
- `occurredAt` is always a valid ISO timestamp in responses.
- `followUpDate`, when present, is a valid date-only value.
- Do not force follow-up date to be on or after occurred date; a user may record overdue follow-up retrospectively.

## Archive

- Archive is soft and timestamp-based.
- Repeated archive requests are idempotent.
- Archived activities are excluded from normal lists and lookups.
- Archived activities cannot be edited through ordinary update.
- No hard-delete endpoint exists.
- Restoration is deferred unless a pre-existing repository convention makes it trivial; do not add it merely for symmetry.
